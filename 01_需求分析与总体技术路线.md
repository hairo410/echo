## 1. 项目概述

### 1.1 目标

交付一台**心理健康AI陪伴机器人原型机**，具备以下能力：

1. **语音对话**：自然语音输入输出，支持打断
2. **基础视觉**：摄像头捕捉 + 云端视觉理解 + 情绪识别
3. **语音量表评估**：通过纯语音对话完成心理量表（PHQ-9/GAD-7等），无需触摸屏或小程序
4. **大模型问诊对话**：基于心理学Prompt的LLM对话
5. **评估摘要生成**：每次对话后自动生成脱敏摘要，供医生参考
6. **数据本地存储**：用户数据存回本地，医生可调阅
7. **LED眼睛表情**：根据对话情绪输出对应表情（基于ASR情绪/视觉情绪二者选一）

### 1.2 硬性约束

| 约束项 | 要求 |
|--------|------|
| 单台BOM成本 | ≤ ¥1,000（不含GPU服务器） |
| 开发周期 | 3个月 |
| 物料偏好 | 国产优先、医疗器械常用优先 |
| 质量体系 | 供应商至少ISO9000 |
| EMC/EMI | 需满足医疗器械相关要求（参考YY 9706.102-2021） |
| 自研程度 | 尽量不自研，优先成熟方案，优先采购已有模块 |
| GPU | 不强求国产GPU，如使用GPU尽量沐曦等国产方案 |
| 交互方式 | 纯语音 + LED眼睛，不额外增加触摸屏/小程序/App |

### 1.3 项目定位

- 本产品为**辅助提示工具**，不作为医疗诊断或治疗建议
- 用户群体：心理咨询场景下的来访者
- 使用场景：咨询室环境（受控电磁环境，Wi-Fi覆盖）

## 2. 需求细化分析

### 2.1 语音对话

| 需求项 | 技术要求 | 优先级 |
|--------|----------|--------|
| 语音唤醒 | 本地离线唤醒，<500ms响应 | P0 |
| 语音识别(ASR) | 中文为主，流式识别，<500ms首字延迟 | P0 |
| 语音合成(TTS) | 中文，自然情感语音，<500ms首音延迟 | P0 |
| 语音打断 | 支持barge-in，用户可打断AI说话 | P0 |
| 方言支持 | 普通话必须，粤语等可选 | P1 |

**关键判断**：ASR/TTS全部走云端API是最简单的实现路径。如对延迟敏感或需离线，SenseVoice + CosyVoice本地部署为备选。具体选型见 [02#2-SenseVoice](02_成熟方案调研与总体选型.md#2-sensevoice-asr语音识别) 和 [02#3-CosyVoice](02_成熟方案调研与总体选型.md#3-cosyvoice-tts语音合成)。

#### 2.1.1 延迟指标来源与行业基准

> **文档中的 `<500ms` 延迟目标是怎么来的？能否达到？**

**行业基准（2024-2025年最新研究）**：

| 来源 | 指标 | 数据 |
|------|------|------|
| Hamming AI (2025，监测400万+次生产语音调用) | TTFW优秀 | **<300ms** |
| Hamming AI | TTFW良好 | **<500ms** |
| Hamming AI | TTFW可接受 | **<800ms** |
| Hamming AI | 每降100ms延迟→CSAT提升~2分 | — |
| HFES 2024学术研究 (Sogemeier et al.) | 最优响应延迟 | **1.5秒**（太短反而不自然） |
| HFES 2024 | 用户不满意阈值 | **5秒**（超过即感知失败） |
| Barthel 2025 (Dagstuhl) | 人类对话轮转间隔 | **200-400ms** |

> 来源：
> - Hamming AI TTFW Benchmark: <https://hamming.ai/glossary/time-to-first-word-ttfw>
> - HFES 2024: Sogemeier et al., *The Importance of Timing*, doi:10.1177/10711813241260290
> - Barthel 2025: *Alexa, you are too slow!* Dagstuhl seminar

**我们能否达到？**

| 环节 | 实测延迟（ESP32-S3 + SenseVoice 本地方案） | 说明 |
|------|------|------|
| 唤醒检测 (WakeNet9) | **110ms** | ESP-SR官方基准，WakeNet9每帧3ms |
| 语音识别 (SenseVoice-Small) | **70ms/10s音频** | 非自回归Encoder-only，比Whisper快15倍 |
| LLM推理 (Qwen-7B-Int4本地) | **310ms** (512 tokens) | 仅本地部署参考；云端百炼API预期500-1000ms |
| TTS合成 (CosyVoice) | **120ms/10s音频** | 优化后，5倍速扩散模型 |
| **端到端总延迟（本地）** | **≈820ms** | 不含网络传输，纯本地 |
| **端到端总延迟（云端API）** | **预估1-2s** | 含WiFi+网络RTT，波动取决于百炼API负载 |

**结论**：
- `500ms` 作为单个环节延迟目标是合理的工程目标
- 端到端延迟 **1-2秒** 在行业认可的最优范围（1.5s）内
- 比5秒不满意阈值有充分余量
- 如对延迟极度敏感，可走本地方案（SenseVoice-Small + CosyVoice本地部署），端到端~820ms

> 详细延迟分解与优化策略见 [04#4.2-性能测试](04_功能实现与开发计划.md#42-性能测试)。

### 2.2 视觉理解与情绪识别

| 需求项 | 技术要求 | 优先级 |
|--------|----------|--------|
| 图像采集 | 200W-500W像素，拍照模式（非实时视频流） | P0 |
| 云端视觉理解 | 上传图片到多模态LLM进行分析 | P0 |
| 情绪识别 | 从面部表情识别基本情绪（高兴/悲伤/愤怒/中性等） | P1 |

#### 2.2.1 摄像头分辨率为什么是200W-500W

**不是越高越好。人脸情绪识别模型只需 48×48 输入，真正瓶颈是JPEG传输带宽。**

| 因素 | 说明 |
|------|------|
| **人脸情绪识别模型** | 标准输入仅 **48×48** 灰度图（FER2013基准）。即使用 dlib/MTCNN 检测人脸也只需 80×80~200×200 像素的face patch |
| **真正瓶颈** | JPEG编码后的文件大小 + Wi-Fi传输带宽。VGA(640×480)一帧约30KB，上传<500ms；UXGA(1600×1200)一帧约80KB，上传需1-2s |
| **200W vs 500W** | 200W(OV2640, 1600×1200)在SVGA分辨率下25FPS完全够用；500W(OV5640, 2592×1944)在720p只有4-6FPS且需要散热片，拍照场景下无优势 |

| 传感器 | 推荐拍照分辨率 | JPEG大小 | 上传时间(WiFi) | 情绪识别是否够用 |
|--------|---------------|----------|---------------|-----------------|
| OV2640 (200W) | SVGA 800×600 | ~30KB | <500ms | ✅ 完全够 |
| OV2640 (200W) | UXGA 1600×1200 | ~80KB | 1-2s | ✅ 静态拍照可用 |
| OV5640 (500W) | 720p | ~60KB | ~600ms | ✅ 但FPS低，拍照无优势 |

> 详细摄像头选型对比见 [03#2.2-摄像头](03_硬件方案与供应链调研.md#22-摄像头ov2640)。

**结论**：200W(OV2640)完全够用。500W不会提升情绪识别精度，反而增加传输延迟和功耗。

#### 2.2.2 云端视觉理解 ≠ 必须用多模态LLM

**情绪识别有更轻更快的专用模型，不需要每次都调用视觉LLM。**

| 方案 | 模型 | 延迟 | 成本 | 准确率 | 适合场景 |
|------|------|------|------|--------|----------|
| **A：专用FER模型（推荐主路径）** | HSEmotion / DeepFace / FERPlus (ONNX) | **<50ms (CPU)** | 免费 | 7情绪 ~70-75% | 纯面部表情识别 |
| **B：多模态视觉LLM** | Qwen3-VL-235B (百炼API) | 1-3s | ¥0.01-0.03/次 | — | 场景理解+情绪描述 |
| **C：混合方案（V1推荐）** | A做表情识别 + B按需做场景理解 | <50ms + 按需1-3s | 极低 | — | 性价比最高 |

**V1推荐**：方案C（混合）
- 情绪识别用 **HSEmotion** 或 **DeepFace** (ONNX runtime，服务端CPU运行，<50ms)
- 仅在需要场景理解时才调用 Qwen3-VL-235B API（如"用户在做什么"、"环境是否安全"）
- 好处：LED眼睛表情反馈从1-3s降到<50ms；节省API费用；大部分请求不需要VL

**三个推荐模型对比**：

| 模型 | 推理时间(CPU) | 准确率 | 安装方式 | 维护方 |
|------|-------------|--------|----------|--------|
| **DeepFace** | ~50ms | ~68% (FER2013) | `pip install deepface` | 社区维护，GitHub 17k+ stars |
| **HSEmotion** | ~30ms | ~66.5% (AffectNet 7-class) | `pip install hsemotion` | HSE University + Sber AI Lab |
| **FERPlus (ONNX)** | ~40ms | ~70% | 需自行下载ONNX模型 | 微软研究院 |

**FER模型准确率业界基准（FER2013数据集，7情绪分类，2024年）**：

| 模型级别 | 准确率范围 | 代表 |
|----------|-----------|------|
| SOTA（学术最佳，可能有过拟合） | 95%（存疑） | CNN-BiLSTM+Attention |
| 实用SOTA（标准划分） | 73-78% | EfficientMobileNet 77.6%、ViT ~74% |
| 轻量级模型（实测可用） | 63-72% | HSEmotion 66.5%、DeepFace ~68%、CLCM 63% |
| **人类水平（参考）** | **~65% (±5%)** | FER2013图像分辨率低，部分标注不准确 |

> FER2013论文: Goodfellow et al., *Challenges in Representation Learning: Facial Expression Recognition Challenge*, ICML 2013  
> FER2013数据集: <https://www.kaggle.com/c/challenges-in-representation-learning-facial-expression-recognition-challenge>  
> DeepFace论文: Taigman et al. (Facebook AI), *DeepFace: Closing the Gap to Human-Level Performance*, CVPR 2014  
> HSEmotion论文 (ABAW-7): Savchenko et al., *Multi-Task Learning and Compound Facial Expression Recognition*, arXiv 2407.13184  
> HSEmotion论文 (ABAW-8): Savchenko et al., *Audiovisual Ambivalence/Hesitancy, Emotional Mimicry Intensity and FER*, arXiv 2503.10399

> 详细实现见 [04#1.4-视觉分析](04_功能实现与开发计划.md#14-摄像头拍照-视觉分析-5人天)。

**关键判断**：
- ESP32-S3端侧做情绪识别不可行（算力不足，精度低）。正确路径：ESP32拍照→上传服务端→服务端跑FER ONNX模型
- 面部表情识别已有大量成熟开源模型（HSEmotion/DeepFace等），不需要依赖多模态LLM
- 视觉LLM仅作为可选补充（场景理解），不是主路径
- ASR文本情绪分析（路径A）与视觉情绪（路径B）可互补，见 [04_功能实现与开发计划.md#15-情绪识别-led眼睛控制-5人天]

### 2.3 语音量表评估

| 需求项 | 技术要求 | 优先级 |
|--------|----------|--------|
| 对话式量表 | 非机械逐题朗读，融入自然对话 | P0 |
| 量表类型 | PHQ-9（抑郁）、GAD-7（焦虑）等标准量表 | P0 |
| 流程管理 | LLM管理对话状态，自动跳题、追问、计分 | P0 |
| 结果输出 | 量表得分 + 简要分析，脱敏后存储 | P0 |

**关键判断**：
- 纯LLM对话式量表可行。不需要单独的小程序或App。
- 实现方式：System Prompt中嵌入量表规则 + 状态机管理（LLM自行维护对话状态）
- 学术验证：IISER Kolkata (2025) 论文已验证LLM对话式PHQ-9/GAD-7评估可行性
- 备选方案：如果LLM做量表不稳定，回退到结构化Prompt + Few-shot的严格管理模式

**结果输出 → 存储的关系**：
- 2.3 的"结果输出"定义量表**产出什么**：结构化得分 + LLM生成的脱敏分析文本
- 2.5 的"数据存储"定义**怎么存、存哪里**：JSON写入服务端本地磁盘，医生通过HTTP接口导出
- 数据流：`LLM产出 → JSON序列化 → data/scales/*.json → 医生导出`
- 量表结果和评估摘要共享同一套存储基础设施，详见 [2.5 数据存储](#25-数据存储) 和 [04_功能实现与开发计划.md#17-数据本地存储-5人天]

### 2.4 大模型问诊对话

| 需求项 | 技术要求 | 优先级 |
|--------|----------|--------|
| 心理咨询对话 | 基于CBT/共情式对话的心理学Prompt | P0 |
| 角色一致性 | 稳定的心理咨询师人设 | P0 |
| 安全边界 | 不声称自己是医生、危机检测与转介提示 | P0 |
| 上下文记忆 | 短期记忆（当前对话）+ 长期记忆（用户档案摘要） | P1 |

**关键判断**：
- 不需要自研模型。使用Qwen3-235B API + 精心设计的System Prompt即可
- 如需蒸馏/微调，使用LlamaFactory + Qwen3-8B做LoRA（成本低，显存需求24GB单卡即可）

### 2.5 数据存储

| 需求项 | 技术要求 | 优先级 |
|--------|----------|--------|
| 对话记录 | 脱敏后存储，文本格式 | P0 |
| 量表结果 | 结构化存储（得分+摘要） | P0 |
| 存储位置 | 服务端本地磁盘（非ESP32 SD卡，SD卡写入不可靠） | P0 |
| 医生调阅 | 通过局域网HTTP接口导出 | P1 |
| 隐私合规 | 人脸图片不存储、语音不存储、仅存脱敏文本 | P0 |

**存储架构概要**：

```
服务器端 (xiaozhi-esp32-server/data/)
├── summaries/           # 评估摘要（JSON，每次对话一份）
│   └── 2025-07-09_1430_session001.json
├── exports/             # 医生导出目录（打包下载）
└── config/              # 系统配置（无实名信息）
```

**不存储**：原始语音录音、原始对话文本、人脸图片、任何可识别身份的信息。

**数据流**：ESP32 → WebSocket音频 → 服务端处理 → LLM生成脱敏摘要 → JSON写入服务端本地磁盘 → 医生通过 `http://服务器IP:8000/export` 下载

> 详细存储架构、数据结构定义、医生导出接口、隐私合规措施见 [04#1.6-摘要](04_功能实现与开发计划.md#16-评估摘要生成-3人天) 和 [04#1.7-存储](04_功能实现与开发计划.md#17-数据本地存储-5人天)。

## 6. 本章结论

### 6.1 推荐方案

**以"小智ESP32-S3"为核心硬件平台，以"xiaozhi-esp32-server"为核心后端，在此基础上增加摄像头拍照、视觉LLM分析、LLM对话式量表、LED眼睛表情、数据本地存储五个功能模块。**

核心硬件选型：立创实战派ESP32-S3（¥148/套）或酷世DIY ESP32-S3-SP-V3（¥119/套）
核心后端：xiaozhi-esp32-server（Docker部署）
核心AI：阿里百炼 Qwen3-235B API

### 6.2 为什么推荐

1. **小智是2024-2025年最活跃的开源AI硬件项目**，GitHub万星，70+开源硬件适配，社区极其活跃
2. **60-80%功能直接复用**，语音对话链路完全不需要开发
3. **新增功能都是"已有方案"的变体**，不是从零设计——摄像头拍照社区已有开源、LED表情有成熟库、LLM量表有学术验证
4. **开发工作量可控**：~51人天，3个月2人团队（1嵌入式+1后端）可完成
5. **供应链成熟**：ESP32-S3、OV2640、MAX7219、ICS-43434均为大批量通用料，淘宝/立创现货充足
6. **不绑定单一供应商**：ESP32可换开发板，LED可换OLED，摄像头可换型号

### 6.3 后续需要继续验证的内容

1. **LLM对话式量表的实际效果**：需用真实量表题目+模拟用户测试PHQ-9/GAD-7语音评估的一致性和准确性
2. **CosyVoice情感TTS的延迟**：需实际测试百炼API的首音延迟是否满足对话体验
3. **小智ESP32固件修改范围**：需详细阅读固件源码，确认拍照+LED表情+SD卡存储修改的可行性
4. **视觉LLM集成**：需确定后端视觉分析handler的具体实现方式（是否复用小智现有的OTA/视觉端口8003）
5. **沐曦GPU实测**：如需部署本地LLM微调版本，需联系沐曦获取测试机/云资源进行兼容性验证
