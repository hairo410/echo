### 1.2 心理学Prompt设计 —— 3人天

**核心工作**：设计System Prompt，不是写代码。

#### Prompt结构

```markdown
# System Prompt (核心设计)

## 角色设定
你是一位专业的AI心理咨询陪伴助手。你具有以下特质：
- 共情但不表演：真诚理解来访者的感受，但不过度表演情绪
- 专业但不冷冰：遵循CBT（认知行为疗法）基本原则，但用温暖自然的语言表达
- 安全边界清晰：当检测到自杀/自伤/他伤风险时，立即启动危机干预协议

## 对话原则
1. 使用开放式问题引导来访者表达
2. 对来访者的情绪给予共情回应（如"听起来你最近承受了很多压力"）
3. 避免直接给建议，而是帮助来访者自己发现解决方案
4. 适时进行认知重构（帮助来访者识别和挑战不合理信念）
5. 每轮对话控制在2-3句话以内，避免大段输出

## 情绪识别与回应
当你在对话中识别到来访者的情绪状态时：
- 用自然的方式label这个情绪（"我注意到你说这些的时候，声音好像有些低落"）
- 不要在每句话中都label情绪，只在情绪变化显著时提及
- 将识别到的情绪以JSON格式记录在内部（不输出给用户）

## 量表评估触发
当用户表示希望做评估或系统触发量表时：
- 自然地将量表问题融入对话（不要机械朗读）
- 严格按照量表规范进行评分
- 完成后生成评估摘要

## 危机干预协议
如果检测到以下任何情况，立即执行：
- 明确的自杀意图或计划
- 对他人造成严重伤害的意图
- 正在发生的自伤行为

响应方式：
"我听到你说的话让我很担心你的安全。请你记住，你不需要一个人面对这些。
你可以拨打24小时心理援助热线：400-161-9995（希望24热线）或 12320（公共卫生热线）。
你愿意现在联系一位你可以信任的人吗？"

## 输出格式
每次回复以纯文本形式输出，最后一行以JSON格式记录内部状态：
{"emotion": "识别到的情绪", "risk_level": "低/中/高", "scale_phase": "非量表/量表中/量表完成", "score": null}
```

### 1.3 LLM对话式量表引擎 —— 10人天（核心新增功能）

这是本项目技术难度最高的部分。

#### 设计思路

不用额外的状态机框架（如LangGraph），而是在System Prompt中嵌入量表规则，让LLM自己管理对话状态。这种方式最简单、最灵活。

#### 量表数据结构（以PHQ-9为例）

```python
PHQ9 = {
    "name": "PHQ-9",
    "title": "患者健康问卷抑郁量表",
    "questions": [
        {"id": 1, "text": "做事时提不起劲或没有兴趣", "options": [
            {"score": 0, "label": "完全没有"},
            {"score": 1, "label": "有几天"},
            {"score": 2, "label": "一半以上天数"},
            {"score": 3, "label": "几乎每天"}
        ]},
        # ... 其余8题
    ],
    "scoring": {
        "0-4": "无抑郁症状",
        "5-9": "轻度抑郁",
        "10-14": "中度抑郁",
        "15-19": "中重度抑郁",
        "20-27": "重度抑郁"
    },

```

#### 量表Prompt（嵌入System Prompt）

```markdown
## 量表评估模块

当触发量表评估时，你需要在对话中自然完成以下量表的评估。

### 量表：PHQ-9
评估过去2周的情况。

问题列表：
1. 做事时提不起劲或没有兴趣
2. 感到心情低落、沮丧或绝望
3. 入睡困难、睡不安稳或睡眠过多
4. 感到疲倦或没有活力
5. 食欲不振或吃太多
6. 觉得自己很糟——或者觉得自己很失败，或者让家人失望
7. 做事时难以集中注意力，例如阅读报纸或看电视
8. 动作或说话速度缓慢到别人已经注意到？或者正好相反——烦躁或坐立不安的情况比平常更严重
9. 有不如死掉或用某种方式伤害自己的念头

评分标准：完全没有=0分，有几天=1分，一半以上天数=2分，几乎每天=3分

### 对话流程
1. 不要逐题朗读。在自然对话中将问题融入，每次只覆盖1-2个维度。
2. 当来访者的回答不够明确时，温和地追问以获得可评分的回答。
3. 在对话结束时，默默计算总分，不要直接告知分数（除非来访者主动询问）。
4. 以自然的方式结束评估对话，例如："谢谢你和我分享这些。从我们刚才的交流中，我对你的状态有了一些了解..."

### 内部状态追踪
每轮对话结束时，在JSON中更新量表状态：
{"scale_name": "PHQ-9", "current_question": 3, "completed_questions": [1,2], "scores": {"1": 2, "2": 1}, "total_so_far": 3}
```

#### 为什么不引入LangGraph等状态机框架

| 方案 | 优势 | 劣势 |
|------|------|------|
| **LLM自管理状态**（推荐） | 极简、灵活、无需额外框架 | 状态可靠性需Prompt工程保证 |
| LangGraph / 状态机 | 状态管理严格 | 引入额外复杂度、与xiaozhi-server架构不兼容 |
| 外部状态机（Python FSM） | 可控性强 | LLM输出解析依赖性强、僵化 |

**结论**：先用LLM自管理状态方案。如果经过测试发现状态管理不稳定（如跳题、重复问），再引入轻量Python状态机辅助。

### 1.4 摄像头拍照 + 视觉分析 —— 5人天

#### 实现架构（更新：双路径混合方案）

```
路径A（主路径 — 面部表情识别）：
[ESP32拍照] → [WebSocket上传JPEG] → [服务端DeepFace/HSEmotion ONNX模型] → [返回情绪标签] → LED表情
延迟：<50ms (ONNX推理) + 传输时间，总延迟<1s

路径B（可选 — 复杂场景理解）：
[ESP32拍照] → [WebSocket上传JPEG] → [百炼Qwen3-VL API] → [返回场景描述+情绪] → 注入LLM对话上下文
延迟：1-3s (API调用) + 传输时间
仅在按键长按或LLM判断需要场景理解时触发
```

**设计决策**：情绪识别不依赖多模态LLM。原因：
- 面部表情识别有成熟的专用小模型（HSEmotion/DeepFace），CPU推理<50ms
- 多模态LLM做表情分类性价比低（1-3s延迟、按token计费）
- 仅在需要场景理解（如"用户在做什么"、"环境是否安全"）时才调用Qwen3-VL

#### 后端新增：双路径处理

```python
# 新增 image_handler.py
from deepface import DeepFace  # pip install deepface
import numpy as np
from PIL import Image
import io

# 路径A：专用FER模型（主路径，每次拍照都执行）
async def analyze_emotion_with_fer(image_bytes):
    """使用 DeepFace 做面部表情识别，CPU <50ms"""
    img = Image.open(io.BytesIO(image_bytes))
    img_array = np.array(img)
    
    # DeepFace分析（内部自动做face detection + emotion recognition）
    result = DeepFace.analyze(
        img_array, 
        actions=['emotion'],
        enforce_detection=False,  # 检测不到人脸也不报错
        silent=True
    )
    
    if result and len(result) > 0:
        emotions = result[0]['emotion']  # {happy: 0.85, sad: 0.03, ...}
        dominant = result[0]['dominant_emotion']  # 'happy'
        return {
            "emotion": dominant,
            "emotion_scores": emotions,
            "face_detected": True
        }
    else:
        return {
            "emotion": "neutral",
            "face_detected": False
        }

# 路径B：多模态VL（可选，仅复杂场景理解时调用）
async def analyze_scene_with_vl(image_bytes):
    """调用Qwen3-VL API做场景理解，1-3s延迟"""
    img_base64 = base64.b64encode(image_bytes).decode()
    response = await aliyun_vl_client.chat(
        model="qwen-vl-max",
        messages=[{
            "role": "user",
            "content": [
                {"type": "image", "image": img_base64},
                {"type": "text", "text": "请用一句话描述场景：用户在哪里、在做什么、环境是否安全。不要描述外观特征。以JSON返回：{scene, activity, environment_safe}"}
            ]
        }]
    )
    return json.loads(response)

# 主handler：路由到路径A（必须）+ 路径B（可选）
async def handle_image_message(websocket, image_data, mode="emotion_only"):
    # 1. 路径A：表情识别（总是执行）
    emotion_result = await analyze_emotion_with_fer(image_data)
    
    # 2. 下发情绪到ESP32控制LED
    await ws_send_json(websocket, {
        "type": "led_emotion",
        "emotion": emotion_result["emotion"],
        "face_detected": emotion_result["face_detected"]
    })
    
    # 3. 路径B：场景理解（仅在mode="scene"时执行）
    scene_info = None
    if mode == "scene":
        scene_info = await analyze_scene_with_vl(image_data)
    
    # 4. 将分析结果注入LLM对话上下文
    context = f"[系统：观察到用户表情为{emotion_result['emotion']}"
    if scene_info:
        context += f"，场景：{scene_info.get('scene')}"
    context += "]"
    # 注入到xiaozhi-server的LLM对话上下文
```

### 1.5 情绪识别 + LED眼睛控制 —— 5人天

#### 情绪识别方案选择

本项目有两种情绪识别路径，择一即可：

**路径A：ASR文本情绪分析（推荐，最简单）**
- LLM在对话中自动识别用户情绪（Prompt引导）
- LLM输出情绪标签→后端转发→ESP32 LED
- 优点：零额外开发，与对话链路共用

**路径B：视觉情绪分析**
- 摄像头拍照→Qwen3-VL分析面部表情→返回情绪标签
- 优点：可以捕捉用户未说出来的情绪
- 缺点：需要用户面对摄像头

**V1推荐**：优先实现路径A（文本情绪），路径B已在1.4（视觉分析）中覆盖，两个路径的结果可以互相补充。

async def after_llm_response(llm_text, websocket):
    """LLM回复后，提取情绪标签并下发LED控制指令"""
    # 从LLM输出的JSON中提取情绪

### 1.6 评估摘要生成 —— 3人天

#### 功能描述

每次心理咨询对话结束后，LLM自动生成一份脱敏摘要，包含：
- 对话主题概要
- 主要情绪状态
- 量表评估结果（如有）
- 建议关注事项
- 风险等级

不包含：
- 用户真实身份信息
- 原始对话文本
- 人脸图片

#### 实现方式

```python
# 在xiaozhi-esp32-server中新增摘要生成模块
async def generate_session_summary(conversation_history, scale_result=None):
    """对话结束后生成评估摘要"""
    
    prompt = f"""
    请根据以下心理咨询对话内容，生成一份脱敏的评估摘要。
    
    ## 对话内容（脱敏后）
    {summarize_conversation(conversation_history)}
    
    ## 量表结果
    {scale_result if scale_result else "本次对话未进行量表评估"}
    
    ## 摘要要求
    1. 简要描述本次对话探讨的主题
    2. 来访者表现出的主要情绪状态
    3. 关键关注点（如有）
    4. 风险等级评估（低/中/高）
    5. 建议（仅供医生参考，不作为诊断）
    
    字数控制在300字以内。不包含任何可识别个人身份的信息。
    """
    
    summary = await llm_client.chat(prompt)
    
    # 存储到本地
    save_summary_to_local(summary, session_id, timestamp)
    
    return summary
```

#### 本地存储方案

ESP32-S3端（SD卡）或服务器端（本地文件/SQLite）：

import json, os
from datetime import datetime

def save_summary_to_local(summary, session_id, timestamp):
    """存储摘要到本地JSON文件"""
    record = {
        "session_id": session_id,
        "timestamp": timestamp,
        "summary": summary,
        "data_type": "deidentified_summary"  # 标记为脱敏数据
    }
    
    # 存储路径（可配置）
    data_dir = os.getenv("SUMMARY_STORAGE_PATH", "./data/summaries")

    filename = f"{timestamp[:10]}_{session_id}.json"
    with open(os.path.join(data_dir, filename), 'w') as f:
        json.dump(record, f, ensure_ascii=False, indent=2)
```

#### 数据流详解

```
[ESP32] --WebSocket音频流--> [xiaozhi-esp32-server]
                                       |
                               [VAD → ASR → LLM → TTS]  ← 实时对话流水线
                                       |
                               [对话结束/量表完成触发]
                                       |

**关键设计决策**：
- **存在服务端而不是ESP32 SD卡**。原因：ESP32 SD卡SPI写入只有~1.8MB/s，且频繁写入有文件系统损坏风险
- **只存脱敏摘要，不存原始数据**。对话文本在生成摘要后丢弃
- **全部JSON格式**，人类可读，医生可直接打开

import zipfile, os
from datetime import date

async def export_patient_data(request):
    """医生导出接口：GET /api/export?from=2025-07-01&to=2025-07-09&key=ACCESS_KEY"""
    
    # 1. 验证访问密钥
    if not validate_key(request.args.get('key')):
        return {"error": "unauthorized"}, 401
    
    # 2. 按日期范围筛选
    from_date = request.args.get('from', '2025-01-01')
    to_date = request.args.get('to', str(date.today()))
    
    # 3. 打包 summaries/ + scales/ 为一个zip
    zip_path = f"data/exports/export_{to_date}.zip"
    with zipfile.ZipFile(zip_path, 'w') as zf:
        for f in filter_by_date(from_date, to_date):
            zf.write(f)
    
    # 4. 返回下载链接
    return {
        "download_url": f"/download/export_{to_date}.zip",
        "file_size": os.path.getsize(zip_path),
        "expires_in": "24h"
    }

# HTTP路由注册

# GET /download/{filename}  → 静态文件下载

#### 不存储的内容

| 内容 | 原因 |
|------|------|
| ❌ 原始语音录音 | 隐私合规，增加存储压力 |
| ❌ 原始对话文本 | 只存脱敏摘要，原文不保留 |
| ❌ 人脸图片 | 分析完即删除，ESP32端不保留帧缓冲 |
| ❌ 姓名/身份证等身份信息 | 系统不采集也不存储 |

#### 隐私合规措施

1. **数据不出机构**：所有数据存储在服务器本地磁盘，不对接任何云端存储（百炼API仅做推理，不存储输入）
2. **医生调阅**：通过局域网HTTP接口 `http://服务器IP:8000/api/export` 下载打包数据
3. **访问控制**：导出接口需要预设的访问密钥（ACCESS_KEY）
4. **自动清理**：`storage_policy.yaml` 配置30天后自动删除，医生导出后本地保留副本
5. **加密**（可选V2）：AES-256加密存储，密钥由机构管理

#### 各环节预期延迟

| 测试项 | 目标 | 实际预期 | 测试方法 |
|--------|------|----------|----------|

| 端到端对话延迟(云端API) | <2s | **1-2s** | 用户说完→听到回复 |

| 拍照→VL场景分析 | <3s | **1-3s** (Qwen3-VL API) | 按键→文本返回 |

#### 延迟优化优先级

1. 如果端到端>2s：先切 TTS 从 CosyVoice API → EdgeTTS（牺牲质量换延迟）
2. 如果 LLM 首 Token >1s：调小 max_tokens，或切到 Qwen3-8B 本地
3. 如果 FER >500ms：检查 DeepFace 模型下载，考虑切换 HSEmotion
4. 如果拍照上传 >1s：降低分辨率到 VGA(640×480)，JPEG quality降到80

> 详细延迟研究数据：Hamming AI TTFW Benchmark <https://hamming.ai/glossary/time-to-first-word-ttfw>
> SenseVoice性能：<https://github.com/FunAudioLLM/SenseVoice>
> ESP-SR基准：<https://docs.espressif.com/projects/esp-sr/en/latest/esp32/benchmark/README.html>

### 4.3 异常场景测试

|------|----------|

## 5. 本章结论

### 5.1 推荐方案

**开发路线**：
1. 先基于小智开源方案跑通语音对话（零开发量）
2. 并行开发三个核心新增功能：量表引擎（后端为主）、摄像头拍照（嵌入式+后端）、LED眼睛（嵌入式为主）
3. 最后整合联调、测试、交付

### 5.2 开发工作量总览

| 类别 | 人天 |
|------|------|
| 直接复用（语音链路） | 0 |
| 配置调整（Prompt等） | 3 |
| 量表引擎（核心） | 10 |
| 摄像头+视觉分析 | 5 |
| LED眼睛+情绪 | 5 |
| 评估摘要+数据存储 | 8 |
| 联调测试 | 10 |
| 外壳+装配 | 5 |
| 文档+交付 | 4 |
| 缓冲 | 7 |
| **合计** | **~57人天** |

### 5.3 为什么推荐这个开发计划

1. **最大程度复用**：语音对话题目零开发，专注在真正新增的功能上
2. **核心优先**：量表引擎排在最前面，这是本项目的核心差异化功能
3. **并行开发**：嵌入式（摄像头/LED）和后端（量表/摘要）可并行推进
4. **缓冲充足**：预留约15%缓冲时间应对意外
5. **3个月可交付**：2.5人 × 12周 ≈ 60人天，匹配57人天的工作量估算

### 5.4 后续需要继续验证

1. **LLM自管理量表状态的可靠性**：这是最大的技术风险，需在W3-4尽早验证
2. **小智固件源码结构**：确认修改ESP32固件的难度（需阅读源码）
3. **社区"双目魔眼"方案的代码质量**：评估可复用程度
4. **百炼Qwen3-VL API的实际延迟**：图片上传+分析的总延迟是否可接受
5. **3D打印外壳与电子件的适配**：结构设计需要实际装配验证
